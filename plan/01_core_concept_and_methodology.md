# 台灣鄉鎮探索導航計畫：核心概念與方法論

## 1. 計畫起源與核心轉變
計畫的初衷源於對台灣流域的探索過程，進而體認到在地探索的本質即是「鄉鎮探索」。

*   **從流域到鄉鎮**： 流域探索雖具特色，但實際落地時，景點與餐飲食宿資訊皆隸屬於特定的鄉鎮行政區劃中，因此將核心調整為「台灣 368 鄉鎮」。
*   **架構前行**： 此計畫強調「先建構框架，再填充內容」。與其逐個點狀記錄，不如先建立涵蓋全台所有鄉鎮的完整架構。
*   **數據與方法論的深度結合**： 實體書不只是紙本紀錄，而是 `WalkGIS (WG)` 數位系統的「操作導引」與「解讀大腦」。

## 2. 三位一體的整合結構 (The Trinity)
計畫明確區分了書本、資料庫與 AI 技能的功能分工，形成一個具備生命力的探索生態系統。

### 2.1 實體書：方法論與解讀大腦
提供探索的「靈魂」：
*   **鄉鎮分類與樣態**： 分析不同鄉鎮的特性（如鬼雷小鄉鎮型、大都市型、原鄉等）。
*   **動態內容引用**： 書本將資料庫中的 POI 視為自己的一部分。當書本提及特定區域時，讀者可透過 SQL 查詢或地圖連結獲取最新數據。
*   **機能發展與規劃**： 從國土規劃維度提供深層解讀。

### 2.2 數位系統 (WG/WIPS)：動態數據中心
利用現有的數位地圖與 SQL 資料庫 (DB)，處理具體的行政區劃與 POI。
*   **Meta Data 整合**： 鄉鎮的分類、Tag 與進度狀態將直接寫入 `walkgis.db` 的 `meta_data` 欄位。
*   **動態列表**： 書本提供對應的 SQL 命令範例，讓使用者能產出即時的統計報表或導航清單。

### 2.3 Antigravity Skill：AI 輔助探索助手
本計畫將隨書附贈專屬的 `antigravity skill`：
*   **Agentic AI 輔助**： 使用者下載 Skill 後，其 AI 代理程式將具備自動操作 `walkgis.db`、生成研究指令與自動撰寫探勘筆記的能力。
*   **自動化流程**： 讓「架構前行」的維護工作（如狀態同步、影像厚化）實現全自動化。

## 3. 鄉鎮分類與標籤系統 (Tagging)
為了處理 368 個鄉鎮的龐大數量，必須導入標籤與分類機制，並直接映射至資料庫。

| 分類維度 | 具體項目說明 |
| :--- | :--- |
| **鄉鎮類型** | 鬼雷小鄉鎮型、大都市型、原鄉等。 |
| **地理位置** | 基於縣市行政區劃進行歸類（如北台灣、溪北地區等）。 |
| **關鍵標籤 (Tags)** | 針對各鄉鎮特色打上 Tag，形成可搜尋的 Tag 清單。 |
| **打卡機制** | 結合農會、車站等特定地標作為鄉鎮抵達的檢核點。 |

## 4. 數據處理進程與執行階段
計畫提出了一套標準化的資料處理流程，將每一個鄉鎮的開發進度視覺化：
1.  **有框架 (Framework)**： 建立初步的行政與地理架構。
2.  **有甜肉 (Content Filling)**： 填充景點、美食等實質內容（AI 厚化）。
3.  **有地圖 (Mapping)**： 完成地理資訊與 W 系統地圖的結合。
4.  **驗證 (Verify)**： 經過實地或進一步的核實確保準確度。

## 5. 技術實現與平台整合 (W 系統)
在本計畫中，**「W 系統」** 是一個核心術語，其完整涵蓋範圍包括：
*   **鄉鎮導航地圖 (The Map)**：在 WalkGIS 平台上運行的「鄉鎮導航」管理圖層。
*   **鄉鎮與 POI 節點**：圖層內包含的全台 368 鄉鎮行政單元及其關聯的景點、美食等數據。
*   **後端資料庫 (walkgis.db)**：儲存所有空間幾何、Meta Data（分類、Tag、狀態）與描述說明的核心資料庫。

*   **系統結合**： 不另建新系統，而是將 368 鄉鎮直接整合進上述的 W 系統中。
*   **屬性欄位 (Attributes)**： 在 W 系統的每個節點中，增加特定欄位（如 `enrichment_status`）來標註處理進度。
*   **算力與資源管理**： 優先建立框架與目錄，再有計畫地分階段引入 AI 生成內容，平衡算力需求。

## 6. 核心觀點與金句
*   **關於架構**：「現在的精神是架構前行嘛，那你既然是一個一個鄉鎮的探索，那你的架構就是整個全部的鄉鎮。」
*   **關於定位**：「書本的目錄主要就是在建構這個方法論……肉（實質內容）就隨著這個 WG 的 map 持續的更新。」
*   **關於順序**：「先寫會經過的鄉鎮，就有點像是倒著寫，一開始架構有，然後就倒著寫。」
